GIT workflow

Noções básicas

A grande maioria dos controles de versão guardam as mudanças do código como alterações de um determinado arquivo. Ou seja, a cada mudança no arquivo, o sistema guarda apenas a mudança e não o arquivo inteiro, o git por outro lado trata tudo como snapshots, sempre que é realizado um commit, o git salva uma referência para um snapshot de como todos os arquivos alterados estão, para os arquivos que não foram modificados ele não cria um novo snapshot, apenas cria uma nova referência para o snapshot antigo.

Tudo no GIT possui um checksum, com isso nada pode ser alterado, criado ou deletado sem seu conhecimento, para o usuário isso significa que é impossível ter um arquivo corrompido ou mesmo perdido sem que o git fique sabendo, é utilizado um mecanismo de hash SHA-1, isso é, uma string de 40 caracteres compostas por caracteres hexadecimais para cálculo de checksum

Estados do GIT

Os arquivos no git estão sempre em um dos três estados fundamentais:

  • committed

    • Arquivos committed são arquivos que já estão versionados e sem modificações pendentes na base local.

  • modified

    • Arquivos modified são arquivos que já estiveram no estado de committed e sofreram alguma alteração.

  • staged

    • Arquivos staged são arquivos que sofreram modificação "modified" e foram marcados para fazerem parte do próximo commit

Seções no GIT


Basicamente o GIT é dividido em 3 grandes áreas de operações:

  • Git directory

    • Local onde o git armazena os metadados e o banco de dados do projeto, quando é feito um clone de um repositório, esse é o local que será clonado

  • Working Directory

    • Local de trabalho, onde os arquivos estão prontos para serem adicionados, editados ou removidos. Esses dados são obtidos através de uma base de dados comprimida do Git directory

  • Stage Area

    • Um único arquivo contendo as informações de arquivos que irão para o próximo commit, também chamado de index.

Pode-se resumir de forma prática o workflow do git como sendo:

  • Clonar o git directory para o working directory
  • Modificar arquivos no working directory
  • Selecionar e adicionar a eles snapshots para staging area
  • Realizar o commit, que levará os arquivos que estão na staging area para o git directory
